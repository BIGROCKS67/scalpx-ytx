Architecture

What we are building

YTX = a standalone ScalpX product - its own route, data model, and UI. One show-lifecycle engine: pick any roster channel · run one checklist through channel setup → pre-show → live → post-show. Same engine for Chento Trades, Banter, and all 9 traders. We pull in what we already built (Whisper, YT import, publish queue, Deals CRM) - we do not extend /studio or bolt onto existing modules.

Deals API: TrackingLink + dealId · sponsor URLs · #ad copy

Convert: tracked links on descriptions when in funnel stage (optional hook)
Remaining ops (DK · not code): 10 real UC channel IDs in data/roster-channel-ids.json · GCP OAuth client (redirect local + Vercel) · YouTube API key in Settings/env · Scout URL + service key · TrackingLinks per deal · connect OAuth on ≥1 channel · run one real show to validate which “Auto” tasks hold.
Not this: extending /studio kanban · hardcoded studio_channel_chento · shared Studio nav. Is this: new product · new entities · new route · optional adapters into existing libs when it saves build time.
